# Правила создания коммитов

## Основной принцип
После выполнения каждой задачи необходимо создавать коммит, который отражает внесенные изменения.

## Шаблон коммита
```bash
git add .
git commit -m "[тип]: краткое описание

- детальное описание изменений"
git push origin main
```

## Типы изменений
- `[feat]` - новая функциональность
- `[fix]` - исправление ошибок
- `[refactor]` - рефакторинг кода
- `[docs]` - изменения в документации
- `[style]` - форматирование, отступы и т.д.
- `[test]` - добавление тестов
- `[perf]` - улучшение производительности
- `[chore]` - обновление зависимостей, настройка сборки и т.д.

## Примеры коммитов

1. Добавление новой функциональности:
```bash
git commit -m "[feat]: добавлен генератор меню

- создан новый сервис для генерации меню
- добавлена валидация входных данных
- интеграция с OpenAI API"
```

2. Исправление ошибок:
```bash
git commit -m "[fix]: исправлена ошибка в расчете калорий

- исправлен баг с неправильным подсчетом калорий
- добавлена проверка на отрицательные значения"
```

3. Рефакторинг:
```bash
git commit -m "[refactor]: оптимизация сервиса покупок

- переработана структура хранения данных
- улучшена производительность алгоритма сортировки
- код разделен на модули"
```

## Правила хорошего коммита

1. Сообщение должно быть информативным
2. Используйте правильный тип изменений
3. Разделяйте краткое и детальное описание
4. Описывайте что и почему изменилось, а не как
5. Группируйте связанные изменения в один коммит
6. Делайте коммиты атомарными (одна логическая единица изменений)

## Процесс создания коммита

1. Проверьте статус изменений:
```bash
git status
```

2. Просмотрите изменения:
```bash
git diff
```

3. Добавьте файлы:
```bash
git add .
```

4. Создайте коммит с информативным сообщением:
```bash
git commit -m "[тип]: краткое описание

- детальное описание"
```

5. Отправьте изменения:
```bash
git push origin main
```

## Важные замечания

1. Коммиты делаются после завершения каждой логической задачи
2. Каждый коммит должен оставлять проект в рабочем состоянии
3. Избегайте слишком больших коммитов с множеством несвязанных изменений
4. Всегда проверяйте работоспособность кода перед коммитом
